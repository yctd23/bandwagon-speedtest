## 搬瓦工测速脚本是什么，为什么要跑

**搬瓦工测速脚本**，就是在 VPS 上跑的一段 Shell 命令，通常可以自动完成以下几件事：系统基础信息采集（CPU 型号、内存、磁盘大小）、硬盘 IO 读写速度测试、从 VPS 向世界各地测速节点的下载带宽测试，以及三网（电信、联通、移动）延迟和路由追踪。

为什么要跑？两个场景会用到。

## 主流测速脚本一览

| 脚本名 | 测试内容 | 适合谁用 |
|--------|---------|---------|
| bench.sh | 系统信息 + 硬盘 IO + 全球节点下载速度 | 快速摸底，最常用 |
| Superspeed.sh | 三网（电信/联通/移动）上传/下载速度 + 延迟 | 专门看国内速度 |
| YABS | 硬盘 IO + 网络带宽 + CPU 跑分（Geekbench） | 需要看 CPU 综合性能 |
| backtrace / NextTrace | 回程路由追踪 | 排查线路质量问题 |

四个脚本不是非此即彼的，通常 bench.sh + Superspeed.sh 组合用，能覆盖大多数人关心的信息。

## bench.sh：最快摸清基础数据

### 为什么先说它

bench.sh 是目前用得最广的 VPS 综合测速脚本，输出结果简洁，五分钟跑完，适合刚到手的 VPS 第一次测试用。

### 命令

bash
wget -qO- bench.sh | bash


一行命令，回车等几分钟就行。

### 读结果

跑完之后会看到几段输出：

- **系统信息段**：CPU 型号、内存大小、虚拟化架构（KVM）、操作系统。搬瓦工用的是 KVM，这里应该显示 KVM。
- **硬盘 IO 段**：会跑三次取平均，一般 SSD 在几百 MB/s 左右属于正常范围。
- **网络测速段**：这是重点。bench.sh 会测 VPS 到世界多个节点的下载速度，包括日本、新加坡、美国、欧洲等地。如果你买的是 CN2 GIA-E 套餐，北美方向的下载通常表现不错。

说实话，bench.sh 的测速节点偏向全球，国内三网测速不是它的强项，这时候就需要配合 Superspeed.sh 一起用。

## Superspeed.sh：专门测三网速度

### 搬瓦工用户最该跑这个

国内用户最关心的是电信、联通、移动三网从 VPS 下载的速度和延迟，Superspeed.sh 正好做这件事。它会把国内各地的测速节点列出来，分省市显示上传速度、下载速度、延迟和抖动，看起来非常直观。

### 命令

bash
bash <(curl -Lso- https://raw.githubusercontent.com/uxh/superspeed/master/superspeed.sh)


网络连通的情况下直接跑，会自动下载并执行。如果碰到权限或者脚本拉取失败，可以先更新一下 apt 再试：

bash
apt-get update && apt-get install -y curl


### 结果怎么看

输出分三段：电信节点、联通节点、移动节点。每个节点显示上传 Mbps、下载 Mbps、延迟 ms、抖动 ms，以及"正常/断流/失败"的状态标注。

一些参考：

- 电信线路，DC6 CN2 GIA-E 机房，白天延迟大概在 130-160ms 之间，晚高峰有时会上去
- 移动线路，搬瓦工走 CMIN2，延迟普遍比电信稍高，但稳定性相对不错
- 联通线路，走 CN2 GIA 或者 CUP，各地差异会有一些

不过数字背后要结合实际用法。只是挂脚本或者建站，延迟高 20ms 其实感受不明显；实时通话或者低延迟游戏加速，就要看得更仔细。

## YABS：加了 CPU 跑分的那个

### 什么情况用

如果除了网速之外，还想知道这台 VPS 的 CPU 性能，YABS 是个不错的选择。它的网络测速用 iperf3，磁盘测试用 fio，CPU 部分调用 Geekbench 跑分，比其他脚本更全面。

缺点是运行时间比较长，完整跑完要 10-20 分钟，Geekbench 那段还会联网下载临时文件。

### 命令

bash
curl -sL yabs.sh | bash


如果只想跑网速和 IO，不想等 Geekbench，可以加参数跳过：

bash
curl -sL yabs.sh | bash -s -- -ig

## backtrace：路由追踪，排查线路问题

有时候速度测试没问题，但实际使用卡顿或者丢包率高。这时候需要看回程路由，也就是数据包从 VPS 回到国内的路径，有没有绕远路或者经过高负载节点。

命令：

bash
curl https://raw.githubusercontent.com/zhanghanyun/backtrace/main/install.sh -sSf | sh


安装后执行 `backtrace` 即可。输出会显示电信、联通、移动的回程路由跳数和对应的 ISP，CN2 GIA 线路的回程应该全程在 CN2 节点上，如果中间出现绕道的情况就值得关注了。

## 测速时几个常见问题

**问：脚本下载失败怎么办？**

大概率是 VPS 和 GitHub 之间的连通性问题。先确认网络没断，然后更新 apt 包管理工具，再重新执行。如果还是不行，可以把脚本内容复制到本地，上传到 VPS 之后直接 `bash 脚本文件名` 执行。

**问：测速结果每次都不一样，正常吗？**

正常。网络状况是动态的，同一台 VPS 白天跑和晚高峰跑结果可能差一截。想看稳定性，多跑几次取均值比单次更有参考价值。

**问：Superspeed.sh 有些节点显示"失败"是出问题了吗？**

不一定。部分测速节点本身稳定性就一般，偶尔断连是正常的。关注电信、联通、移动各自**大部分节点**的整体表现，单个节点失败不用太在意。

**问：搬瓦工可以一键换机房吗？**

CN2 GIA-E 套餐可以，在 KiwiVM 控制面板里直接操作，无需额外付费，数据随迁。这也是为什么很多人选 CN2 GIA-E 的原因之一——测速发现某个机房不满意，换一个再测，很灵活。

**问：测速脚本安不安全？**

这是个合理的顾虑。主流脚本（bench.sh、Superspeed.sh、YABS）的 GitHub 仓库都是公开的，可以在跑之前把脚本内容看一遍。执行前建议先确认命令来源可信，不要盲目运行陌生渠道的 curl 命令。

## 关于测速机房的几个实际经验

测速结果有时候跟直觉不一样。

比如软银机房，日间延迟很低，但晚高峰电信方向有时候波动比较大，如果你主要在工作时间用，体感可能比 CN2 GIA 还好，但熬夜刷视频就容易感受到差别。

再比如荷兰联通 EUNL_9 机房，在 CN2 GIA-E 套餐里可以切过去，联通用户测速经常会有惊喜，电信就一般。三大运营商用什么机房其实各有侧重。

讲真，测速脚本的数字只是参考，实际用的场景——建站、挂机、SSH 开发、还是跑代理——对速度要求的维度完全不一样。把脚本当成初筛工具用，结合实际用途做判断才是正确姿势。
